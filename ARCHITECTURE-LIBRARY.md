# Crucible — Model Library Architecture

> Companion to ARCHITECTURE-FORGE.md and ARCHITECTURE-PORTAL.md.
> Describes the Model Library: theory catalog, ensemble builder, and
> recommendation engine.

---

## What it is

The Model Library gives consultants (Forge) and clients (Portal) a
structured way to understand, browse, and apply the 25 academic theories
that power Crucible simulations. It serves three use cases:

1. **Discovery** — "What models exist? What do they do?"
2. **Curation** — "Which models are right for this scenario?"
3. **Ensemble building** — "Let me customize the theory mix before running."

## Backend components

### `api/catalog.py` — Theory catalog extractor

Introspects the theory registry at startup. No changes to theory modules.

```
TheoryCatalogEntry
  theory_id         str
  display_name      str              (snake_case → Title Case)
  domains           list[str]        (from cls.DOMAINS)
  short_description str              (first line of module docstring)
  full_description  str              (full module docstring)
  academic_reference str             (last "Reference: ..." line)
  parameters        list[ParameterInfo]
    .name           str
    .description    str              (from Field(description=...))
    .default        Any
    .min_val        float | None     (ge/gt constraint)
    .max_val        float | None     (le/lt constraint)
    .type_name      str
  reads             list[str]        (state_variables.reads)
  writes            list[str]        (state_variables.writes)
  initializes       list[str]        (state_variables.initializes)
```

Catalog is built once at startup, cached in memory. No DB needed — the
theory modules ARE the source of truth.

### `api/routers/theories.py` — Theory catalog endpoints

```
GET  /api/theories
     → list[TheoryCatalogEntry.summary()]
     query params: domain=macro&domain=market (multi-select filter)
                   q=bass  (keyword search in name + description)

GET  /api/theories/{theory_id}
     → TheoryCatalogEntry.detail()

POST /api/theories/recommend
     body: { scenario_text: str, domain: str | None }
     → RecommendResponse
          recommendations: list[{
            theory_id: str,
            display_name: str,
            reasoning: str,      # one sentence from Claude
            confidence: "high" | "medium" | "low",
            priority: int,
            suggested_parameters: dict[str, Any]
          }]
          method: "domain_match" | "claude" | "hybrid"
```

Recommendation strategy (hybrid):
1. **Domain match** (fast, deterministic) — map scenario domain tag to
   a pre-computed set of theory IDs. Returns instantly.
2. **Claude refinement** (async) — pass scenario description + domain
   match results to Claude (claude-haiku-4-5, cheap + fast). Returns
   ranked list with one-sentence reasoning per theory and suggested
   parameter overrides. Replaces domain match result if successful.
3. If Claude call fails, domain match result is returned with
   `method: "domain_match"`.

Domain → theories default map (seeds the fast path):
```python
DOMAIN_MAP = {
    "geopolitics":   ["richardson_arms_race", "fearon_bargaining",
                      "wittman_zartman", "keynesian_multiplier"],
    "market":        ["porter_five_forces", "cournot_oligopoly",
                      "cobweb_market", "bass_diffusion"],
    "macro":         ["keynesian_multiplier", "is_lm", "solow_growth",
                      "minsky_instability"],
    "technology":    ["bass_diffusion", "experience_curve",
                      "schumpeter_disruption", "fisher_pry"],
    "ecology":       ["lotka_volterra", "hotelling_cpr", "sir_contagion"],
    "conflict":      ["richardson_arms_race", "fearon_bargaining",
                      "wittman_zartman", "opinion_dynamics"],
    "corporate":     ["porter_five_forces", "principal_agent",
                      "efficiency_wages", "stackelberg_leadership"],
}
```

### `api/routers/ensembles.py` — Ensemble CRUD

```
GET    /api/ensembles
       → list[EnsembleSummary]  (id, name, theory_ids, created_at)

POST   /api/ensembles
       body: { name: str, theories: list[TheoryRef], source: "user"|"system" }
       → EnsembleSummary

GET    /api/ensembles/{id}
       → Ensemble (full, including TheoryRef list with parameters)

DELETE /api/ensembles/{id}
       → 204

POST   /api/ensembles/{id}/fork
       body: { name: str }
       → new EnsembleSummary (copy with new id)
```

Storage: `data/ensembles/{id}.json`
No DB required for MVP. Migrate to PostgreSQL in Week 10 if multi-user.

## Data flow: intake → ensemble → simulation

```
User types scenario description
        │
        ▼
POST /forge/intake  (existing scoping agent)
POST /api/theories/recommend  (fires in parallel)
        │                            │
        ▼                            ▼
SimSpec draft builds         Recommendations arrive
(actors, timeframe, etc.)    (streaming or polled)
        │                            │
        └──────────────┬─────────────┘
                       ▼
            EnsembleBuilder rendered
            (recommended panel pre-populated)
                       │
              Consultant customizes
                       │
                       ▼
            POST /api/ensembles  (save if named)
                       │
                       ▼
            POST /simulations  (SimSpec + ensemble merged)
                       │
                       ▼
            Dashboard (existing)
```

## Relationship to existing architecture

| Component | Relationship |
|-----------|-------------|
| `core/theories/` | **Read-only source** — catalog extracts metadata, never modifies |
| `core/spec.py` — `TheoryRef` | **Ensemble output format** — selected theories become `TheoryRef[]` in SimSpec |
| `forge/scoping_agent.py` | **Handoff point** — scoping agent fires `/api/theories/recommend` when domain is known |
| `ARCHITECTURE-FORGE.md` | **Extended** — intake flow gains ensemble step between SimSpec completion and sim launch |
| `ARCHITECTURE-PORTAL.md` | **Extended** — client portal gains read-only "Models used" panel (Phase 3) |

## What's NOT in scope (MVP)

- Theory parameter sensitivity explorer (run sim with parameter sweep)
- Theory dependency graph visualization (which theories read each other's outputs)
- User-contributed theory modules (plugin marketplace)
- Per-theory performance benchmarks / accuracy scores
- Version history for ensembles (fork covers most of this need)
